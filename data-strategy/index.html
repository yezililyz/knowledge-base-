XVentures Data Strategy

XVentures connects investors, founders, regulators, and corporates at professional events by turning raw registration data into precision-ranked meeting shortlists. This document explains the data architecture — from raw form fields to shortlisted pairs with meeting-type-specific explanations.

Why a Data Strategy?

Every event has structured participant data: roles, mandates, product readiness, compliance capabilities, investment theses. Without a systematic strategy, this data sits unused in form fields or requires expensive manual curation per event.

With XVentures' Data Strategy

Organizers collect identifiers only
A registration form captures role, goals, and freetext. XVentures automatically enriches each profile with structured tags, readiness scores, and mandate summaries — no manual tagging required.
Every match has business meaning
Shortlisted pairs come with specific explanations grounded in the attendee's actual mandate: thesis, regulatory program, product stage, compliance capability. No generic copy.
Two meeting types surface independently
Capital meetings (P1) and operational meetings (PILOT) are computed independently. A regulator-founder pair that would score P3 in a single-dimension system correctly surfaces as PILOT.
No implementation complexity
Upload a CSV or connect via API. The pipeline enriches, scores, and ranks automatically. Attendees receive shortlists ranked from their own perspective using persona-aware weights.

How It Works

A catalog in XVentures consists of a reconciliation key (attendee_id), enriched profile fields, and pairwise scoring logic. The pipeline runs four agents in sequence, each with a focused input and clean JSON output contract.

The Two Readiness Dimensions

When creating a match, XVentures evaluates readiness across two fully independent dimensions. This determines which meeting type a pair is eligible for — a core architectural change in v2.1.

DIMENSION 1
Financial Readiness
FinancialReady(A,B)
MIN(fin_ready_A, fin_ready_B)
Best for
✓ P1 bucket (≥ 0.70): capital meetings
✓ Investor thesis matching
✓ Round size alignment
Example: Elena (investor, fin=0.91) + James (founder, fin=0.85) → MIN=0.85 → P1 eligible.
DIMENSION 2
Operational Readiness
OperationalReady(A,B)
MIN(ops_ready_A, ops_ready_B)
Best for
✓ PILOT bucket (≥ 0.65): sandbox/pilot meetings
✓ Regulator-founder collaboration
✓ Corporate integration partnerships
Example: Sophie (Bundesbank, ops=0.91) + James (ops=0.88) → MIN=0.88 → PILOT eligible.

Critical rule: FinancialReady and OperationalReady are never blended. A pair with financial_ready=0.10 and operational_ready=0.85 is a PILOT match, not P3. Keeping the two dimensions independent prevents capital-deployment metrics from obscuring sandbox readiness.

The Four-Agent Pipeline

Each agent has a single job, a defined input schema, and a clean JSON output contract. Agent 1 runs once per attendee and caches its output. Agents 2–4 run per directed pair and can be parallelised.

Processing Report

Each import generates a processing report in the organiser dashboard: enrichment_quality distribution, needs_review count, persona override rate, most common data_gaps, and preview of the first 10 enriched profiles. The report is differentiated by import mode (API, sFTP, manual).

FAQ

Why does a Series B company sometimes score lower traction than a Seed company? +
Stage label is never a quality signal — it is a filter only. Traction is scored exclusively from evidence: institutional clients, audit history, TVL, known investors, regulatory approvals. A Seed company with enterprise clients and a Trail of Bits audit will outscore a Series B company with no demonstrable traction.
Why does a pair appear in PILOT even though neither side is raising capital? +
FinancialReady and OperationalReady are fully independent dimensions. PILOT fires when match_score ≥ 0.70 AND operational_ready ≥ 0.65 AND financial_ready < 0.70. This surfaces regulator-founder and corporate-founder pilot opportunities that would be incorrectly buried in P3 under a single-dimension system.
What happens if actively_raising is checked but freetext says 'we recently closed our Series B'? +
Agent 1 sets actively_raising_confirmed=false. The contradiction overrides the checkbox. fin_ready_score is reduced — the raising_signal component drops to 0.0 instead of 1.0. The mismatch is logged in the processing report.
Why are rank_scores directional? Why isn't A→B the same as B→A? +
Because each persona values readiness differently. An investor with alpha=0.60 weights the founder's financial readiness heavily. The founder's rank_score for the same investor uses alpha=0.50 and gamma=max — a different formula producing a different rank. Each person's shortlist is ranked from their own perspective.
When does a profile get flagged needs_review=true? +
When persona_confidence < 0.70 (self-reported persona contradicts evidence), enrichment_quality < 0.30 (insufficient signal — missing LinkedIn, thin freetext), or when the persona override confidence gap exceeds 0.20.
What is the difference between thematic_fit and complementary_score? +
thematic_fit measures topic alignment: do they work in the same verticals, regulatory environments, geographies? complementary_score is a weighted aggregate of persona_fit + intent_overlap + thematic_fit — it measures structural match quality holistically. A pair can have high thematic_fit but low complementary_score if their intents conflict.
When does non_obvious_score matter most? +
When vertical tag overlap is low (<0.25) but mandate_summary semantics suggest similar underlying problems. This surfaces founders from different sectors solving the same compliance or infrastructure challenge. When tag_overlap ≥ 0.50, non_obvious_score is capped at 0.10 — the match is obvious, not a discovery.

Glossary

No terms match ""
Reconciliation key
The shared identifier between registration data and enriched profile. In XVentures: attendee_id.
fin_ready_score
Individual financial readiness (0–1). Founders: from raising/timeline/size signals. Investors: deploy signal + EOL risk. Regulators/corporates: constant 0.20.
ops_ready_score
Individual operational readiness (0–1). From product liveness, sandbox intent, pilot timeline, and compliance depth. Investors: constant 0.20.
FinancialReady(A,B)
Pairwise = MIN(fin_ready_A, fin_ready_B). A pair is only as financially ready as its least-ready member.
OperationalReady(A,B)
Pairwise = MIN(ops_ready_A, ops_ready_B). Fully independent of FinancialReady — never blended.
match_score
Weighted sum of six components. Weights are persona-specific from the event playbook. Range 0–1.
rank_score
Directional: match_score × (1 + α × deal_cpty + β × deal_self). A→B ≠ B→A.
Snapshot context
Profile data frozen at enrichment time. Used for historical analysis, A/B testing of event formats, compliance audit trails.
Update context
Profile data updated when the attendee edits their registration. For late-registration corrections and last-minute profile changes.
P1 bucket
Priority capital meeting. Requires match_score ≥ 0.70 AND financial_ready ≥ 0.70.
PILOT bucket
Sandbox or partnership meeting. Requires match_score ≥ 0.70 AND operational_ready ≥ 0.65 AND financial_ready < 0.70.
P2 bucket
Strategic meeting. Requires match_score ≥ 0.65 AND MAX(financial_ready, operational_ready) ≥ 0.40.
P3 bucket
Post-event follow-up. Requires match_score ≥ 0.40.
mandate_summary
One sentence (50–250 chars) naming ≥1 specific signal. Feeds intent_overlap and non_obvious_score via semantic comparison.
jaccard overlap
Tag similarity: |A∩B| / |A∪B|. Used for vertical, regulatory, geography, and compliance tags.
non_obvious_score
High when mandate semantics align despite low tag overlap (<0.25). Capped at 0.10 when tag_overlap ≥ 0.50 (obvious match).
enrichment_quality
Confidence score (0–1) for each enriched profile. Triggers needs_review=true when < 0.30.
fund_eol_risk
clamp((years_elapsed - 5) / 5, 0, 1). A 10-year-old fund = 1.0 risk (end of typical lifecycle).
gamma
Persona parameter selecting readiness dimension for rank_score. 'fin'=financial, 'ops'=operational, 'max'=MAX of both.
alpha / beta
Playbook weights for counterparty readiness (alpha) and self-readiness (beta). Investor: α=0.60, β=0.05. Regulator: α=0.20, β=0.05.
